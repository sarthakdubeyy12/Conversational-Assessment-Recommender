PHASE 12: PRODUCTION CONVERSATION ORCHESTRATOR
==============================================

OVERVIEW
--------
Production-grade conversation orchestrator that serves as the central workflow
engine, coordinating all existing components without containing any business logic.
This is the "brain" of the AI Agent that executes the complete workflow state machine.

ARCHITECTURE
-----------
State Machine Workflow:

START
  ↓
STATE RECONSTRUCTION
  ↓
INTENT DETECTION
  ↓
GUARDRAILS CHECK
  ↓
DECISION ROUTING
  ├─→ BLOCKED (if guardrails fail)
  ├─→ NEEDS_RETRIEVAL
  │     ↓
  │   RETRIEVAL PIPELINE
  │     ↓
  │   DECISION ROUTING
  │     ├─→ RECOMMENDATION ENGINE
  │     └─→ COMPARISON ENGINE
  │
  ├─→ NEEDS_CLARIFICATION
  └─→ GENERATING_RESPONSE
        ↓
      COMPLETED

COMPONENTS
----------

1. Domain Layer (src/orchestrator/domain/)
   - pipeline_stage.py
     * PipelineStage: Enum of all orchestration stages
     * STATE_RECONSTRUCTION, INTENT_DETECTION, GUARDRAILS, etc.
   
   - execution_context.py
     * ExecutionContext: Mutable context carrying data through pipeline
     * Accumulates results from each stage
     * Contains user message, history, configuration, results
   
   - execution_trace.py
     * ExecutionTrace: Complete audit trail of execution
     * StageTrace: Individual stage execution record
     * Tracks timing, status, errors, metadata
     * Audit-ready and compliance-friendly
   
   - orchestration_result.py
     * OrchestrationResult: Complete output from orchestrator
     * OrchestrationStatistics: Performance metrics
     * Contains response, success flag, traces, component results

2. Workflow Layer (src/orchestrator/workflow/)
   - workflow_state.py
     * WorkflowState: State machine states
     * START, RETRIEVING, RECOMMENDING, COMPARING, COMPLETED, etc.
   
   - decision_router.py
     * Routes workflow based on intent and guardrails
     * Deterministic routing logic
     * No business logic, pure coordination
     * Methods:
       - route_after_guardrails() → next state based on guardrails
       - route_after_intent() → next state based on intent type
       - route_after_retrieval() → next state based on results

3. Execution Layer (src/orchestrator/execution/)
   - pipeline_executor.py
     * Executes individual pipeline stages
     * Delegates to existing engines (never contains business logic)
     * Methods for each stage:
       - execute_state_reconstruction()
       - execute_intent_detection()
       - execute_guardrails_input()
       - execute_retrieval()
       - execute_recommendation()
       - execute_comparison()
     * Updates context and trace for each stage
   
   - error_handler.py
     * ExecutionError: Custom error with recovery info
     * ErrorHandler: Graceful error handling
     * Generates fallback responses
     * Determines if pipeline can continue
     * Stage-specific fallback messages

4. Response Layer (src/orchestrator/response/)
   - response_builder.py
     * Assembles final responses from pipeline results
     * Methods:
       - build_recommendation_response()
       - build_comparison_response()
       - build_clarification_response()
       - build_greeting_response()
       - build_refusal_response()
       - build_fallback_response()
     * User-friendly formatting
     * No business logic

5. Engine Layer (src/orchestrator/engine/)
   - conversation_orchestrator.py
     * Main orchestrator class
     * Executes complete workflow state machine
     * Pure coordination, no business logic
     * Key method: execute(user_message, history, session_id)
     * Returns OrchestrationResult
   
   - orchestrator_factory.py
     * Dependency injection factory
     * Wires up all components
     * Single point of creation

WORKFLOW EXECUTION
------------------
The orchestrator follows this exact flow:

1. Initialize Context & Trace
   - Generate execution ID
   - Create execution context
   - Start audit trace

2. State Reconstruction
   - Delegate to ConversationStateEngine
   - Reconstruct state from history
   - Update context with state

3. Intent Detection
   - Delegate to IntentEngine
   - Detect user intent
   - Update context with intent

4. Guardrails Check
   - Delegate to GuardrailsEngine
   - Validate input safety
   - Block if violation detected

5. Decision Routing (based on intent)
   - RECOMMENDATION → Retrieval → Recommendation Engine
   - COMPARISON → Retrieval → Comparison Engine
   - CLARIFICATION → Generate clarification request
   - GREETING → Generate greeting
   - PROMPT_INJECTION/REFUSAL → Block with refusal

6. Retrieval (if needed)
   - Delegate to RetrievalPipeline
   - Search knowledge base
   - Update context with results

7. Recommendation/Comparison (based on route)
   - Delegate to RecommendationEngine or ComparisonEngine
   - Generate structured output
   - Update context with results

8. Response Generation
   - Delegate to ResponseBuilder
   - Format user-friendly response
   - Return final result

DESIGN PRINCIPLES
-----------------

1. Stateless Execution
   - Every request reconstructs conversation state
   - No session caching
   - No memory between requests

2. Pure Coordination
   - Zero business logic in orchestrator
   - Delegates everything to engines
   - Only workflow control

3. Error Recovery
   - Graceful degradation
   - Fallback responses
   - Continue when possible

4. Dependency Injection
   - All components injected
   - No direct instantiation
   - Easy testing and mocking

5. Comprehensive Tracing
   - Every stage tracked
   - Timing metrics collected
   - Audit-ready output

ORCHESTRATOR NEVER DOES
------------------------
❌ Recommend assessments
❌ Compare assessments
❌ Retrieve vectors
❌ Validate catalog
❌ Perform semantic search
❌ Generate prompts
❌ Contain prompt templates
❌ Contain business logic

ORCHESTRATOR ALWAYS DOES
-------------------------
✅ Coordinate components
✅ Execute state machine
✅ Route decisions
✅ Handle errors
✅ Track metrics
✅ Generate audit trails
✅ Format responses

USAGE EXAMPLE
-------------
```python
from src.orchestrator.engine.orchestrator_factory import OrchestratorFactory

# Create orchestrator with all dependencies
orchestrator = OrchestratorFactory.create_production_orchestrator(
    state_engine=state_engine,
    intent_engine=intent_engine,
    guardrails_engine=guardrails_engine,
    retrieval_pipeline=retrieval_pipeline,
    recommendation_engine=recommendation_engine,
    comparison_engine=comparison_engine,
)

# Execute workflow
result = await orchestrator.execute(
    user_message="I need to hire a software engineer",
    conversation_history=[],
    session_id="session-123",
)

# Access results
print(result.response)  # User-facing response
print(result.success)   # True/False
print(result.statistics.total_duration_ms)  # Performance
print(result.execution_trace.to_dict())     # Audit trail
```

PERFORMANCE METRICS
-------------------
Typical execution times (with 15 doc KB):
- State Reconstruction: 5-10ms
- Intent Detection: 10-20ms
- Guardrails: 1-3ms
- Retrieval: 50-100ms
- Recommendation: 5-15ms
- Comparison: 0.5-2ms
- Total: 70-150ms (without LLM)

Stage Distribution:
- Retrieval: ~60-70% of time
- Intent Detection: ~10-15%
- State Reconstruction: ~5-10%
- Other stages: ~10-20%

ERROR HANDLING
--------------
Recoverable Errors:
- Intent detection failure → clarification
- Retrieval failure → fallback message
- Recommendation failure → request more info
- Comparison failure → clarification

Non-Recoverable Errors:
- State reconstruction failure → critical error
- Guardrails failure → critical error

All errors generate user-friendly messages.

INTEGRATION
-----------
The orchestrator is now the central entry point for the AI Agent.
Future FastAPI endpoints will call:
  orchestrator.execute(user_message, history, session_id)

This provides a clean separation between:
- API layer (FastAPI) → presentation
- Orchestration layer (Orchestrator) → workflow
- Business logic layer (Engines) → domain logic
- Infrastructure layer (DB, Vector Store) → data
