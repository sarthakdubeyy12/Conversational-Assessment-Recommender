================================================================================
OVERVIEW
================================================================================

Phase 6 implements a production-grade Conversation State Engine that reconstructs
complete hiring context from stateless conversation history.

This is NOT a chatbot - it's a state management layer that understands what the
user wants to hire, extracting and organizing that information for downstream
recommendation and comparison modules.

KEY PRINCIPLE: API is stateless. State is reconstructed from messages every time.

================================================================================
KEY COMPONENTS
================================================================================

1. CONVERSATION STATE
   - Complete hiring context (role, skills, experience, etc.)
   - Conversation status (initial, gathering, ready, etc.)
   - Metadata (completeness, confidence, missing info)
   - Validation results

2. REQUIREMENT EXTRACTOR
   - Pattern-based extraction from messages
   - Extracts: role, seniority, skills, experience
   - Keyword matching for assessment needs
   - Handles natural language variations

3. CORRECTION HANDLER
   - Detects correction signals ("actually", "instead", "change to")
   - Replaces old values with corrections
   - Tracks correction history
   - Latest message wins

4. CONTEXT INFERRER
   - Infers seniority from experience/role
   - Infers assessment needs from role
   - Infers leadership from seniority
   - Marks inferred vs explicit fields

5. CONFLICT DETECTOR
   - Detects logical contradictions
   - Junior + 10 years experience
   - Senior title + junior level
   - Entry level + leadership required

6. COMPLETENESS CHECKER
   - Identifies missing information
   - Prioritizes by importance
   - Calculates completion percentage
   - Weighted field scoring

7. STATE BUILDER
   - Orchestrates full reconstruction pipeline
   - Combines extraction, inference, validation
   - Builds complete ConversationState
   - Determines conversation status

8. STATE ENGINE
   - Main API entry point
   - Stateless reconstruction
   - Deterministic (same input → same output)
   - Fast (<100ms typical)

================================================================================
CONVERSATION STATE STRUCTURE
================================================================================

ConversationState:
  hiring_context:
    # Core requirements
    role_title: str | None
    seniority: str | None  # junior, mid, senior, executive
    years_of_experience: int | None
    industry: str | None
    employment_type: str | None
    
    # Skills
    required_skills: List[str]
    preferred_skills: List[str]
    technical_skills: List[str]
    soft_skills: List[str]
    
    # Assessment preferences
    assessment_types_requested: List[str]
    languages: List[str]
    remote_requirement: bool | None
    
    # Specific requirements
    leadership_required: bool | None
    personality_required: bool | None
    cognitive_required: bool | None
    coding_required: bool | None
    
    # Constraints
    must_have: List[str]
    nice_to_have: List[str]
    
    # Free-form
    job_description: str | None
    user_goals: List[str]
  
  status: ConversationStatus
    # INITIAL
    # GATHERING_INFORMATION
    # CLARIFICATION_REQUIRED
    # READY_FOR_RETRIEVAL
    # READY_FOR_RECOMMENDATION
    # REFINEMENT
    # COMPARISON
    # COMPLETED
  
  metadata:
    conversation_id: str | None
    message_count: int
    last_updated: datetime
    inferred_fields: Set[str]
    explicit_fields: Set[str]
    confidence_score: float  # 0.0 to 1.0
    completion_percentage: float  # 0.0 to 1.0
    missing_information: List[str]
    clarification_history: List[Dict]
    detected_conflicts: List[str]
  
  is_valid: bool
  validation_errors: List[str]

================================================================================
STATE RECONSTRUCTION PIPELINE
================================================================================

Input: List[Message]

Pipeline:
1. Extract requirements from messages
   - Parse user messages for hiring signals
   - Extract role, skills, experience
   - Normalize values

2. Detect and apply corrections
   - Find correction signals
   - Replace old values
   - Track corrections

3. Infer missing context
   - Infer seniority from experience/role
   - Infer assessment needs
   - Mark inferred fields

4. Check completeness
   - Identify missing fields
   - Calculate completion percentage
   - Prioritize missing information

5. Detect conflicts
   - Check seniority vs experience
   - Check role vs seniority
   - Flag contradictions

6. Validate state
   - Validate required fields
   - Check value ranges
   - Generate errors

7. Determine status
   - Based on completeness
   - Based on conflicts
   - Based on readiness

Output: ConversationState

USAGE
================================================================================

from src.conversation.domain.entities import Message
from src.conversation.state.state_engine import ConversationStateEngine

# Initialize engine
engine = ConversationStateEngine()

# Example 1: Single message
messages = [
    Message(role="user", content="I need to hire a senior Java developer")
]

state = engine.reconstruct_state(messages)

# Access hiring context
print(state.hiring_context.role_title)  # "Senior Java Developer"
print(state.hiring_context.seniority)   # "senior"
print(state.status.value)               # "gathering_information"
print(state.metadata.completion_percentage)  # 0.3

# Example 2: Progressive conversation
messages = [
    Message(role="user", content="Hiring backend engineer"),
    Message(role="assistant", content="What level?"),
    Message(role="user", content="Senior, 7 years experience"),
    Message(role="assistant", content="What skills?"),
    Message(role="user", content="Python, Kubernetes, leadership"),
]

state = engine.reconstruct_state(messages)

print(state.hiring_context.role_title)         # "Backend Engineer"
print(state.hiring_context.seniority)          # "senior"
print(state.hiring_context.years_of_experience) # 7
print(state.hiring_context.technical_skills)   # ["Python", "Kubernetes"]
print(state.hiring_context.leadership_required) # True
print(state.status.value)                      # "ready_for_recommendation"

# Example 3: Correction
messages = [
    Message(role="user", content="Need Java developer"),
    Message(role="assistant", content="Okay..."),
    Message(role="user", content="Actually, make that Python developer"),
]

state = engine.reconstruct_state(messages)

print(state.hiring_context.role_title)  # "Python Developer" (not Java!)

# Example 4: Check readiness
if state.is_ready_for_recommendation():
    print("Ready to retrieve assessments!")
    
if state.needs_clarification():
    print("Missing info:", state.get_missing_fields())
    print("Conflicts:", state.get_conflicts())

# Example 5: Serialize state
state_dict = state.to_dict()
# Can be stored, logged, or passed to other modules

================================================================================
CONVERSATION STATUS DETERMINATION
================================================================================

INITIAL:
  - No role title extracted yet
  - Very beginning of conversation

GATHERING_INFORMATION:
  - Has role but missing important info
  - Completion < 40%

READY_FOR_RETRIEVAL:
  - Has minimum information for search
  - Completion >= 40%

READY_FOR_RECOMMENDATION:
  - Has sufficient information
  - Completion >= 60%
  - <= 2 missing fields
  - No conflicts

CLARIFICATION_REQUIRED:
  - Detected conflicts
  - Needs user to resolve

REFINEMENT:
  - User refining recommendations

COMPARISON:
  - User comparing assessments

COMPLETED:
  - Conversation finished

================================================================================
DESIGN PRINCIPLES
================================================================================

1. STATELESS
   - No server-side session storage
   - Reconstruct from messages every time
   - Enable horizontal scaling

2. DETERMINISTIC
   - Same messages → same state
   - No randomness
   - Reproducible results

3. PURE FUNCTIONS
   - No side effects
   - Testable
   - Composable

4. SEPARATION OF CONCERNS
   - Extraction separate from inference
   - Inference separate from validation
   - Each module has one job

5. PROGRESSIVE DISCLOSURE
   - Minimum viable state first
   - Infer carefully
   - Ask for critical info only

6. EXPLICIT > IMPLICIT
   - Mark inferred fields
   - Track explicit fields
   - Don't confuse the two

7. LATEST WINS
   - Corrections replace old values
   - Most recent message has priority
   - Natural conversation flow
