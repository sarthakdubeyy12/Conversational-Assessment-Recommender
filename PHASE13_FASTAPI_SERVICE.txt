PHASE 13: PRODUCTION FASTAPI SERVICE
=====================================

OVERVIEW
--------
Production-grade FastAPI service providing HTTP API endpoints for the
Conversational SHL Assessment Recommender. The API layer is EXTREMELY THIN,
containing zero business logic - it only handles HTTP transport concerns.

DESIGN PRINCIPLES
-----------------

1. **Thin API Layer**
   - NO business logic whatsoever
   - Only HTTP concerns: validation, serialization, status codes
   - All business logic in orchestrator and engines

2. **Exact Schema Compliance**
   - Request/response schemas match assignment spec exactly
   - Contract tests verify compliance
   - Never deviate from specification

3. **Security First**
   - Never expose internal errors
   - Never expose stack traces
   - Never expose system details
   - User-friendly error messages only

4. **Dependency Injection**
   - All dependencies injected via FastAPI Depends()
   - Singleton orchestrator (lru_cache)
   - Clean separation of concerns

5. **Production Ready**
   - Comprehensive error handling
   - Structured logging
   - Health checks
   - OpenAPI documentation

EXACT API CONTRACT
------------------

**POST /chat**

Request Schema:
```json
{
  "messages": [
    {"role": "user", "content": "I need to hire a software engineer"},
    {"role": "assistant", "content": "What skills are you looking for?"},
    {"role": "user", "content": "Python and problem solving"}
  ]
}
```

Rules:
- `messages`: Array of 1+ messages (REQUIRED)
- Each message has `role` ("user" or "assistant") and `content` (non-empty string)
- Last message MUST be from "user"
- Content cannot be empty or whitespace-only

Response Schema:
```json
{
  "reply": "Based on your requirements, I recommend...",
  "recommendations": [
    {
      "title": "Verify G+",
      "url": "https://www.shl.com/solutions/products/assessments/verify-gplus/",
      "description": "Measures general cognitive ability",
      "competencies": ["Problem Solving", "Analytical Thinking"],
      "duration": "45 minutes"
    }
  ],
  "end_of_conversation": false
}
```

Rules:
- `reply`: Non-empty response text (REQUIRED)
- `recommendations`: Array of 0-10 recommendations (REQUIRED, can be empty)
- `end_of_conversation`: Boolean flag (REQUIRED, default false)
- Each recommendation has `title` and `url` (required), others optional

Response Codes:
- 200: Success
- 422: Validation error (malformed request)
- 500: Internal error (orchestrator failure)

COMPONENTS
----------

### 1. Request Models (src/api/models/chat_request.py)

**MessageModel**
- Represents a single conversation message
- Fields: role (Literal["user", "assistant"]), content (str)
- Validates content is not empty

**ChatRequest**
- Complete chat endpoint request
- Fields: messages (List[MessageModel])
- Validates:
  * At least one message
  * Last message from user
  * Content not empty
- Methods:
  * get_current_message() → last user message
  * get_history() → all messages except last

### 2. Response Models (src/api/models/chat_response.py)

**RecommendationModel**
- Single assessment recommendation
- Fields:
  * title (str) - REQUIRED
  * url (str) - REQUIRED
  * description (Optional[str])
  * competencies (Optional[List[str]])
  * duration (Optional[str])

**ChatResponse**
- Complete chat endpoint response
- Fields:
  * reply (str, min_length=1) - REQUIRED
  * recommendations (List[RecommendationModel], max 10) - REQUIRED
  * end_of_conversation (bool) - REQUIRED
- Validates max 10 recommendations

### 3. Health Models (src/api/models/health_response.py)

**HealthResponse**
- Service health status
- Fields: status, version, environment

### 4. Error Models (src/api/models/error_response.py)

**ErrorResponse**
- Generic error response
- Fields: error (str), detail (Optional[str])
- Never exposes internals

**ValidationErrorResponse**
- Validation error (422)
- Fields: error (str), detail (List[ErrorDetail])
- Contains field-level errors

**ErrorDetail**
- Single validation error
- Fields: loc (List[str]), msg (str), type (str)

### 5. Chat Route (src/api/routes/chat.py)

Endpoint: POST /chat
Handler: async def chat(request, orchestrator)

Flow:
1. Extract current message and history from request
2. Convert to orchestrator format (ConversationMessage)
3. Call orchestrator.execute()
4. Check success flag
5. Extract recommendations (max 10)
6. Determine end_of_conversation:
   - True if COMPLETION intent
   - True if REFUSAL intent
   - True if blocked by guardrails
   - False otherwise
7. Build ChatResponse
8. Return response

Error Handling:
- HTTPException(500) if orchestrator fails
- All other errors caught by global handler
- Never expose internal details

### 6. Health Route (src/api/routes/health.py)

Endpoint: GET /health
Handler: async def health()

Returns:
- status: "healthy"
- version: "1.0.0"
- environment: from settings

### 7. Error Handler (src/api/middleware/error_handler.py)

Handles:
- RequestValidationError → 422 with field errors
- ValidationError → 422 with field errors
- Exception → 500 with generic message

Never exposes:
- Stack traces
- Internal error details
- System information
- File paths

### 8. Orchestrator Dependency (src/api/dependencies/orchestrator.py)

**get_orchestrator()**
- Creates singleton orchestrator
- Uses @lru_cache for singleton pattern
- Wires up all dependencies:
  * KnowledgeBaseRetriever
  * CatalogService
  * StateEngine
  * IntentEngine
  * GuardrailsEngine
  * RetrievalPipeline
  * RecommendationEngine
  * ComparisonEngine
- Returns configured ConversationOrchestrator

Dependencies:
- All engines from Phase 1-12
- Settings from environment
- Logging infrastructure

### 9. Application Factory (src/api/app.py)

**create_app()**
- Creates FastAPI application
- Configures CORS middleware
- Adds logging middleware
- Adds exception handlers
- Registers routes (health, chat)
- Sets up lifecycle hooks (startup, shutdown)
- Returns configured app

WORKFLOW
--------

### Chat Request Flow

1. **Request Received**
   - FastAPI receives POST /chat
   - Pydantic validates request body
   - Raises 422 if validation fails

2. **Dependency Injection**
   - get_orchestrator() called
   - Returns singleton orchestrator instance
   - All engines already initialized

3. **Request Processing**
   - Extract current message (last in array)
   - Extract history (all except last)
   - Convert to ConversationMessage format

4. **Orchestrator Execution**
   - Call orchestrator.execute(message, history)
   - Orchestrator runs complete workflow:
     * State reconstruction
     * Intent detection
     * Guardrails check
     * Retrieval (if needed)
     * Recommendation/Comparison (if needed)
     * Response generation
   - Returns OrchestrationResult

5. **Result Processing**
   - Check result.success flag
   - Extract recommendations (if present)
   - Limit to 10 recommendations max
   - Determine end_of_conversation flag

6. **Response Building**
   - Create ChatResponse with:
     * reply (from result.response)
     * recommendations (from result.recommendation_result)
     * end_of_conversation (from intent/guardrails)
   - Pydantic validates response
   - Return 200 with JSON

7. **Error Handling**
   - If orchestrator fails → 500
   - If validation fails → 422
   - All errors logged but never exposed

CONVERSATION END CONDITIONS
---------------------------

The `end_of_conversation` flag is set to True when:

1. **Completion Intent**
   - User says "thank you", "goodbye", etc.
   - Intent engine detects COMPLETION

2. **Refusal Intent**
   - User explicitly refuses to continue
   - Intent engine detects REFUSAL

3. **Guardrails Block**
   - Prompt injection detected
   - Off-topic conversation
   - Safety violation
   - Guardrails should_block() returns True

Otherwise, flag is False (conversation continues).

INTEGRATION WITH PHASE 12
--------------------------

The FastAPI service is the HTTP entry point that delegates to
the orchestrator from Phase 12:

```
HTTP Request
    ↓
FastAPI Endpoint (POST /chat)
    ↓
Request Validation (Pydantic)
    ↓
Dependency Injection (get_orchestrator)
    ↓
Orchestrator Execution (Phase 12)
    ↓
    ├─→ State Reconstruction (Phase 4)
    ├─→ Intent Detection (Phase 7)
    ├─→ Guardrails Check (Phase 11)
    ├─→ Retrieval Pipeline (Phase 8)
    ├─→ Recommendation Engine (Phase 9)
    └─→ Comparison Engine (Phase 10)
    ↓
Orchestration Result
    ↓
Response Building
    ↓
Response Validation (Pydantic)
    ↓
HTTP Response (200/422/500)
```

Clean separation:
- FastAPI: HTTP concerns only
- Orchestrator: Workflow coordination
- Engines: Business logic
- Infrastructure: Data access
